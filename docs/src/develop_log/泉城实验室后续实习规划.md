# 泉城实验室后续实习规划

我想以参加2026年OS内核赛道国赛的方式继续在泉城实验室做开源项目实习。不仅通过这次比赛提升了自己的os开发能力也能让我把有趣并且现实的想法落地的能力

我这次打算在我的内核并且结合文件系统，动态驱动加载这方面做扩展：

## 一、LVM 块设备聚合

现在axfs的存储，没有跨盘管理的能力。我想在驱动层和文件系统层之间加一个逻辑卷管理层，把多块物理盘聚合成一个逻辑存储池，上层文件系统不需要关心底下有几块盘。


## 二、DAX 直接内存映射

os的普通的文件读写要经过内核缓冲区，数据在内核和用户空间之间来回拷贝。DAX 的思路是让进程直接把存储设备的某段区域映射到自己的地址空间，读写直接打到设备上，省掉中间的拷贝。

需要在 axfs-ng 的 VFS 接口上加 DAX 支持，配合内存管理模块做页表映射。

这两块加在一起会呈现 LVM 把多块盘合成一个逻辑空间，DAX 让进程可以直接 mmap 这个逻辑空间，底层跨了几块盘对上层完全透明，然后后期还可以对硬盘进行动态扩容实现更灵活的存储管理。

## 三、通用驱动动态加载框架

我想按照linux的ko动态驱动加载的思想实现一个.ko 的动态驱动按需加载的方式（从王一丁同学的实现和linux结合想到的）驱动都放在文件系统里，内核启动时扫描，需要某个能力时再加载对应驱动。

核心想法是在王一定同学的ondemand上继续扩展，定义一套驱动描述规范：每个驱动声明自己能提供什么能力，内核需要某个能力时向框架请求，框架找到对应驱动加载进来。这套规范如果做成独立 crate，其他 OS 也能复用。

## 整体流程

```text
内核启动 → 框架扫描驱动列表，建立能力索引 -> os kernel检查驱动加载(在需要驱动的地方写一行请求)->
需要块设备 → 框架检查最初扫描的驱动列表是否有在对应能力位有申明->从硬盘加载对应驱动 → 驱动注册物理块设备
LVM 层接管多个物理块设备 → 对外暴露逻辑卷
进程 mmap 逻辑卷（DAX）→ 直接映射到物理存储，零拷贝（绕过page cache）
```
